# Security Policy

## Responsible Use

This penetration testing tool is designed for **authorized security testing only**.

### ✅ Acceptable Use

- Testing systems you own
- Testing with explicit written permission from the system owner
- Bug bounty programs with defined scope
- Authorized penetration testing engagements
- Educational use in controlled lab environments
- CTF (Capture The Flag) competitions
- Security research with proper authorization

### ❌ Prohibited Use

- Testing systems without authorization
- Unauthorized access to computer systems
- Disrupting or damaging target systems
- Denial of Service (DoS) attacks
- Testing production systems without permission
- Bypassing authentication without authorization
- Data exfiltration without permission
- Any illegal activities

## Legal Considerations

### United States
- **Computer Fraud and Abuse Act (CFAA)** - Unauthorized access is a federal crime
- Penalties include fines and imprisonment

### European Union
- **Network and Information Systems Directive (NIS)**
- **GDPR** - Data protection regulations apply

### United Kingdom
- **Computer Misuse Act 1990**
- Unauthorized access is illegal

### Other Jurisdictions
- Most countries have similar laws against unauthorized computer access
- Always verify local laws before conducting security testing

## Best Practices

### 1. Authorization
- **Always obtain written permission** before testing
- Define the scope clearly in writing
- Keep authorization documents accessible during testing
- Verify authorization periodically for long-term engagements

### 2. Scope Management
- Only test systems within the defined scope
- Respect IP ranges and domain restrictions
- Do not pivot to out-of-scope systems
- Document all systems tested

### 3. Responsible Disclosure
- Report vulnerabilities to the appropriate party
- Allow reasonable time for remediation
- Follow coordinated disclosure timelines
- Do not publicly disclose before fixes are available

### 4. Data Handling
- Do not access sensitive data unnecessarily
- Do not download or exfiltrate data without permission
- Securely delete any sensitive data collected during testing
- Follow data protection regulations

### 5. Testing Ethics
- Respect rate limits to avoid service disruption
- Test during agreed-upon time windows
- Stop testing if systems become unstable
- Document all actions for audit trails

### 6. Evidence Collection
- Keep detailed logs of all testing activities
- Screenshot or record proof of vulnerabilities
- Store evidence securely
- Provide clear reproduction steps

## Reporting Security Issues

If you discover security issues in this tool itself:

1. **Do not** open a public GitHub issue
2. Email the maintainers directly at [security email]
3. Include detailed reproduction steps
4. Allow 90 days for response and patching

## Bug Bounty Programs

When using this tool for bug bounties:

### Preparation
- Read and understand the program rules
- Check the scope carefully
- Note any prohibited testing methods
- Verify the rewards and payment terms

### During Testing
- Stay within the defined scope
- Follow the program's testing guidelines
- Do not test production systems if prohibited
- Respect the program's communication channels

### Reporting
- Use the program's submission process
- Provide clear, detailed reports
- Include proof of concept
- Suggest remediation steps
- Be professional and respectful

### Common Bug Bounty Rules
- No DoS/DDoS attacks
- No social engineering
- No physical testing
- No testing of out-of-scope systems
- No spam or excessive automated testing
